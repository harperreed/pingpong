# GoReleaser Release Pipeline Migration — Design

- **Date:** 2026-06-26
- **Status:** Pending user review
- **Branch:** `goreleaser-migration`

## Context

`pingpong-rs` ships a Homebrew formula to `harperreed/homebrew-tap`, the same tap
that serves Harper's other tools (`toki`, `position`, `chronicle`, …). Those
formulae are all **generated by GoReleaser** and use the proven `.tar.gz`
archive pattern.

pingpong is the odd one out. Its release pipeline (`.github/workflows/release.yml`)
hand-rolls the Homebrew formula via `scripts/update-homebrew-formula.sh` and uses
the deprecated `actions/create-release@v1` + `actions/upload-release-asset@v1`
actions. More importantly, **the Homebrew tap has never worked**: the
`publish-homebrew` job was added in commit `18268a7`, which is not contained in
any released tag. The Release workflow only fires on `push: tags: 'v*'`, and the
last tag (v0.1.6) predates that commit — so the job has never run, and
`Formula/pingpong.rb` was never pushed to the tap. `brew install harperreed/tap/pingpong`
fails because the formula does not exist.

This work migrates pingpong's release pipeline to GoReleaser, bringing it in line
with the rest of the tap and producing a working Homebrew formula.

## Goals

1. **Working Homebrew tap** — `brew install harperreed/tap/pingpong` installs a
   working `pingpong` command on macOS (Intel + Apple Silicon).
2. **Consistency** — pingpong's formula is generated by GoReleaser with the same
   `.tar.gz` pattern as the other tools in the tap.
3. **Retire the hand-rolled pieces** — delete `scripts/update-homebrew-formula.sh`
   and the deprecated `create-release@v1` / `upload-release-asset@v1` actions.
4. **Keep crates.io publishing** — `cargo install pingpong-rs` continues to work.
5. **Unify the command name** — the binary is `pingpong` everywhere, including
   `cargo install`.

## Non-Goals

- **No Linux or Windows release binaries.** Scoped out per decision below. The
  Homebrew formula is macOS-only (matching the existing tap formulae), and
  non-mac users install via `cargo install pingpong-rs`.
- **No restructuring of `ci.yml`.** Its cross-platform build matrix is CI
  validation, a separate concern, and stays as-is. The one unavoidable touch: the
  `pingpong` binary rename means its `binary-name` values change from `pingpong-rs`
  to `pingpong` (otherwise its artifact-upload paths break). That is a required
  consequence of the rename, not new scope.
- **No re-architecture of the app.** This is release tooling only.

## Decisions (resolved during brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Approach | Migrate fully to GoReleaser | User chose "GoReleaser migration only" |
| Release scope | **macOS + crates.io only** | Brew (mac) and `cargo install` are the real install paths; drop Linux/Windows binaries |
| Build tool | **Native `cargo`, no zig** | macOS-only on a macOS runner needs no cross-compile; zig adds nothing at this scope (YAGNI) |
| Binary name | **Unify on `pingpong`** via `[[bin]] name = "pingpong"` | Command is `pingpong` everywhere incl. `cargo install` |

### Feasibility validated

Before committing to the design, every target was cross-compiled locally from an
arm64 Mac (rustc 1.95.0):

- `cargo zigbuild` produced clean binaries for `x86_64-unknown-linux-gnu`,
  `x86_64-unknown-linux-musl`, `x86_64-pc-windows-gnu`, and `x86_64-apple-darwin`
  (surge-ping / ICMP linked fine on all). This proves zig is viable if Linux/Windows
  are ever re-added.
- Plain `cargo build --release` (no zig) produced both `aarch64-apple-darwin` and
  `x86_64-apple-darwin` binaries. **This is the path the design uses** — no zig
  needed for a macOS-only build on a macOS runner.

## Architecture

The pipeline triggers on `push: tags: 'v*'` and runs two independent jobs.

### Job 1 — `goreleaser` (macos runner)

1. `actions/checkout@v4` with `fetch-depth: 0` (GoReleaser needs full history for
   the changelog).
2. `dtolnay/rust-toolchain@stable` and add the `x86_64-apple-darwin` target
   (the runner is arm64; cross to x86_64 mac is native via the Apple toolchain).
3. `goreleaser/goreleaser-action@v6` with `distribution: goreleaser` (the
   open-source build, not Pro), `version: '~> v2.16.0'` (pinned — see the
   "Update (2026-06-26)" section below), args `release --clean`.

GoReleaser then, driven by `.goreleaser.yaml`:
- builds the two darwin binaries with native `cargo`,
- packages each as a `.tar.gz` archive,
- generates a checksums file,
- **creates the GitHub release** and uploads the archives + checksums,
- writes `Formula/pingpong.rb` to `harperreed/homebrew-tap`.

Env: `GITHUB_TOKEN` (release creation, auto-provided) and `TAP_GITHUB_TOKEN`
(push to the tap repo, already configured as a repo secret).

### Job 2 — `publish-crate` (ubuntu runner)

Runs in parallel with Job 1. Publishes to crates.io using `CARGO_REGISTRY_TOKEN`
(already configured). GoReleaser has no native crates.io support, so this stays a
small separate job. Reuses `katyo/publish-crates@v2` — the action already used by
this repo's pipeline — which publishes only when the version is new, so a re-run
or a tag that bumps nothing does not fail.

## New file — `.goreleaser.yaml`

```yaml
version: 2

project_name: pingpong

builds:
  - id: pingpong
    builder: rust
    binary: pingpong
    tool: cargo
    # The Rust builder's `command` defaults to `zigbuild`; force plain
    # `cargo build` so no cargo-zigbuild/zig toolchain is needed on the runner.
    command: build
    flags:
      - --release
    targets:
      - x86_64-apple-darwin
      - aarch64-apple-darwin

archives:
  - id: pingpong
    formats: [tar.gz]
    # GoReleaser renders Go-style arch (amd64/arm64); map amd64 -> x86_64 to
    # match the existing tap asset names (Darwin_x86_64 / Darwin_arm64).
    name_template: 'pingpong_{{ .Version }}_Darwin_{{ if eq .Arch "amd64" }}x86_64{{ else }}{{ .Arch }}{{ end }}'

checksum:
  name_template: "checksums.txt"

brews:
  - name: pingpong
    repository:
      owner: harperreed
      name: homebrew-tap
      token: "{{ .Env.TAP_GITHUB_TOKEN }}"
    homepage: "https://github.com/harperreed/pingpong"
    description: "A beautiful TUI ping utility for monitoring network connectivity"
    license: "MIT"
    test: |
      system "#{bin}/pingpong", "--help"

release:
  github:
    owner: harperreed
    name: pingpong
```

Exact field names (e.g. `formats` vs `format`, `brews` vs `homebrew_casks`) are
pinned against the installed GoReleaser v2 during implementation; the generated
formula is diffed against `toki.rb` / `position.rb` to confirm it matches the
existing tap style. Note GoReleaser's Rust builder converts the target triple to
Go-style arch, so `{{ .Arch }}` renders `amd64`/`arm64`; the `name_template`
above maps `amd64` back to `x86_64` so asset names match the rest of the tap.

## Cargo.toml change

Add an explicit binary target so cargo produces a `pingpong` binary for all build
paths (cargo build, cargo install, GoReleaser):

```toml
[[bin]]
name = "pingpong"
path = "src/main.rs"
```

The crate name on crates.io stays `pingpong-rs` (that is `[package].name`, which
is unchanged). The only behavior change: `cargo install pingpong-rs` now installs
a `pingpong` command instead of `pingpong-rs` — a consistency fix, since the
command has always been `pingpong` everywhere else.

## Deletions

- `scripts/update-homebrew-formula.sh` — replaced by GoReleaser `brews`.
- The entire current `release.yml` body (the `create-release`, `build-release`
  matrix, and `publish-homebrew` jobs) — replaced by the two jobs above. This
  removes the deprecated `actions/create-release@v1` and
  `actions/upload-release-asset@v1`.

## Secrets (all already configured)

- `GITHUB_TOKEN` — auto-provided by Actions.
- `TAP_GITHUB_TOKEN` — push to `harperreed/homebrew-tap` (added 2026-06-26).
- `CARGO_REGISTRY_TOKEN` — crates.io publish.

## Verification plan

1. **Local dry run (gate before any tag):**
   `goreleaser release --snapshot --clean` builds both binaries and generates
   `Formula/pingpong.rb` into `dist/` **without publishing anything**. Inspect the
   generated formula and diff it against `toki.rb` for shape.
2. **`brew audit`/`brew style`** on the generated formula.
3. **Real release:** push `v0.1.7`. Confirm the GitHub release has the two
   `.tar.gz` archives + checksums, and that `Formula/pingpong.rb` lands in the tap.
4. **End-to-end:** `brew tap harperreed/tap && brew install pingpong` and run
   `pingpong --help` from the brew-installed binary.
5. **crates.io:** confirm v0.1.7 publishes and `cargo install pingpong-rs` yields
   a working `pingpong`.

## Risks

- **GoReleaser Rust builder maturity.** Stable since v2.5 but newer than the Go
  path. Mitigated by the macOS-only / native-cargo scope (the lowest-risk
  configuration) and the local `--snapshot` gate.
- **Binary-name detection.** GoReleaser's Rust builder must locate the cargo
  output and honor `binary: pingpong`. The `[[bin]] name = "pingpong"` change
  makes cargo's own output `pingpong`, eliminating any rename ambiguity.
- **First release is the real test.** The `publish-homebrew` path cannot be fully
  exercised without a tag; the `--snapshot` dry run + formula diff de-risk it as
  far as possible beforehand.

## Update (2026-06-26): `brews` deprecation

During implementation, the local snapshot revealed that GoReleaser **hard-deprecated
the `brews:` key as of v2.16** (soft since v2.10). Its replacement, `homebrew_casks`,
generates a Homebrew **Cask**, not a Formula. The snapshot confirmed `brews:` still
works in v2.16 and produces a `pingpong.rb` matching `toki.rb`'s shape exactly, but a
floating `version: '~> v2'` would eventually pull a GoReleaser release that removes
`brews:` and silently re-break the tap — the exact failure this migration exists to fix.

**Decision (user-approved):** keep the Formula (`brews:`) and **pin the GoReleaser
version to `~> v2.16.0`** (the 2.16.x patch line) in `release.yml`, rather than switch
to a Cask. Rationale: a Cask would make pingpong the lone Cask in an all-Formula tap
and, because the binary is unsigned/un-notarized, would require a post-install hook to
clear Gatekeeper quarantine — friction a Formula avoids. Migrating the whole tap to
`homebrew_casks` (with signing/notarization) is left as a separate, deliberate effort.
The release pipeline never runs `goreleaser check`, so that command's non-zero exit on
the deprecation does not affect the pipeline.
